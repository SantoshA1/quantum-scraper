# INV-A Decision — Observability vs Functional

**Date:** 2026-05-30 (Saturday close-out)
**Question:** Is the 100% RISK_UNKNOWN pattern on `quantum.exec_flow_audit` since 2026-05-19 17:51:12 UTC an (A) observability bug (audit-only) or (B) a functional bug (Risk Gate actually broken in production)?

**Verdict: (A) Observability bug.**

**Rationale (one paragraph):** Section 2 disproved schema/writer/query-level breakage — the Risk Gate Postgres query returns a valid row when executed manually, all 5 referenced columns exist with the right types, and the writer to `quantum.position_risk_state` is healthy with a write within the last 24 hours. Section 3 located the actual defect in `QTP Early Exec Flow Audit Builder` line 115, which writes a hardcoded sentinel `risk_gate_decision = 'RISK_UNKNOWN'` that is meant to be patched by `QTP Late Audit Patcher` downstream of `Alpaca Paper Trade`; any execution that terminates before reaching Alpaca (regime conflict, extended-hours gate, VC kill, bias block, pause guard block) leaves the sentinel intact. Empirically, in a 10-execution sample, 0/10 reached Alpaca and 10/10 left a sentinel row. UNKNOWN-4 sealed it: on 2026-05-19 — the only day with non-sentinel rows in the past 30 days — every one of the 14 `RISK_PASS` BUY rows correlates 1:1 with a real Alpaca order in `quantum.order_events` (broker_order_id present, status PARTIALLY_FILLED or FILLED), and every one of the 9 `RISK_BLOCK` / `RISK_HOLD` SELL rows correlates with **zero** order activity. The Risk Gate was functionally healthy on the only day it was visible. There is no evidence of functional impact on order placement at any point.

**Implication for Monday remediation:** The fix is a single Code-node edit in `QTP Early Exec Flow Audit Builder` (see `proposed_audit_builder_fix.diff`, risk 2/5). It introduces a new audit enum value `RISK_NOT_REACHED` for upstream-blocked candidates, preserving idempotency with the Late Patcher via its existing `WHERE risk_gate_decision IN ('RISK_UNKNOWN','UNKNOWN','')` filter. **A functional bug fix (rebuilding the Risk Gate Postgres path, adding fail-closed defaults around the Postgres node, etc.) would be the wrong fix and would introduce risk where there is currently none.**

**What remains unexplained:** Why 2026-05-19 17:51:12 UTC was the specific cutoff. We have no `ssm_workflow_updates` audit coverage for that day (earliest row is 2026-05-26 20:37:50 UTC). Hypothesis to test Monday: a workflow edit changed the wiring of the Late Patcher around that time. Check git history of any SSM workflow JSON export from 2026-05-19.
